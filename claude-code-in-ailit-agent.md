# Применение подхода Claude Code в ailit-agent

Практическое руководство: **что перенимаем как идею**, **где лежит реализация в ailit**, **чего не ждать** (лицензии, форматы Anthropic «как есть»).

Оглавление документации проекта: [`docs/INDEX.md`](docs/INDEX.md). Стратегический контекст: [`plan/ailit-global-agent-teams-strategy.md`](plan/ailit-global-agent-teams-strategy.md).

---

## 1. Принцип: архитектура, а не копия репозитория

Код `claude-code` использовать как **референс слоёв и UX-паттернов** (глобальный дом конфига, merge источников, teams, плагины как низший слой defaults). Реализация в `ailit-agent` — своя, под Python, merge в `project_layer` и CLI `ailit`.

---

## 2. Глобальный конфиг и проектные переопределения

| Идея Claude Code | В ailit |
|------------------|---------|
| Дом настроек пользователя (`~/.claude`, `CLAUDE_CONFIG_DIR`) | Канонический глобальный каталог (см. этапы G–I в стратегии), переменные окружения для override |
| Слои: user → project → local → flags | Единый merge resolver: global → project → dev YAML → CLI |
| Секреты не в git | Проектные файлы с gitignore; ключи через env / secure storage по политике команды |

**Практика:** настраивать провайдера через `ailit config`, не править YAML как единственный путь (этап H).

---

## 3. Чат против агента

| Режим | Цель пользователя | Ожидаемое поведение ailit |
|--------|-------------------|----------------------------|
| `ailit chat` | Диалог, обзор проекта, file tools | Human-readable основной поток; JSON/JSONL — в диагностике |
| `ailit agent` | Сценарии, CI, воспроизводимость | Стабильный машиночитаемый поток событий + понятная сводка usage (этап O) |

---

## 4. Инструменты и промпт

- File tools и явные инструкции «не утверждать запись без вызова инструмента» — в духе Claude Code (строгая связка намерение → tool → результат).
- Политика `suppress_tools_after_write_file` и лимиты `max_turns` — **продуктовый** слой (как у агентских клиентов), не путать с `max_tokens` API.

---

## 5. Teams и плагины (дорожная карта)

- **Teams:** mailbox на ФС + tool отправки + наблюдаемость в UI — см. этап L в стратегии; референс структуры каталогов в стратегии §3.
- **Плагины:** MVP — импорт skills и манифеста в `.ailit/plugins/<id>`, не «запуск бинарников Claude» (этап M).

---

## 6. Наблюдаемость и стоимость

- Считать и показывать **usage** нормализованно (input/output/reasoning/cache read/write где доступно) — этап O; ориентир по подаче метрик: OpenCode (`tokens`, `cache.read` / `cache.write` в витрине).
- Лимит шагов сессии (`max_turns`) документировать и дублировать подсказками в UI (этап N).

---

## 7. TUI и мульти-контекст (следующий крупный UX-слой)

- **TUI** с чат-подобным циклом и slash-командами (`/help`, `/model`, …) — этап P; изучать паттерны Claude Code и OpenCode CLI.
- **Killer feature:** несколько **именованных контекстов** в одном TUI, переключение, отдельный учёт токенов — этап Q.

---

## 8. Чек-лист для разработчика ailit

1. Изменение затрагивает merge конфига — проверить и `ailit chat`, и `ailit agent run` на одном проекте.
2. Новый пользовательский вывод — не смешивать сырой wire-format моделей с человекочитаемым текстом (санитайзация нормализации + презентеры).
3. Любой новый инструмент — описание для модели + human-readable презентер результата в чате.
