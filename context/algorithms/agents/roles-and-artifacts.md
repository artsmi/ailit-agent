# Роли 02–09 и 11–13: handoff, артефакты и запреты (Roles and artifacts)

> **Аннотация:** сводная матрица ролей мультиагентного пайплайна под **01** (без отдельной роли исследования: исследование для канона — **`start-research`**): что возвращает каждая роль (JSON и markdown), кто потребляет результаты, какие записи **запрещены**, и как это связано с **OR-001** (границы и handoff) и **OR-002** (нет запуска Subagents у **02+**). Деталь машины состояний — в [`pipeline-topology.md`](pipeline-topology.md).

## Status

`draft`

## Исходная цель (в пересказе)

Дать **одну таблицу опоры** для постановок и ревью: кто владеет каким артефактом, в каком порядке роли читают выход предыдущих, и где проходит граница между кодом продукта и каноном **`context/**`**.

## Why This Exists

Без явной матрицы исполнители смешивают **план артефактов 06** с **`plan/`** репозитория, вкладывают канон в чат **01** вместо **100**, или пытаются получить исследование для канона из сессии **01** вместо входа **`start-research`** (**101–108**).

## Связь с исходной постановкой

**OR-001** требует обзора ролей **02–09** и **11–13** под **01** (и самого **01** как оркестратора): границы, handoff JSON/markdown, гейты. Этот документ закрывает **ролевую матрицу** и запреты; граф процесса и терминальные состояния — в [`pipeline-topology.md`](pipeline-topology.md).

**OR-002** требует явной оркестрации и запретов для **02+**. Здесь в колонке **Subagents** зафиксировано: только **01** (и вне этого файла — **100** для другого контура) имеют право запуска; все строки **02–09** и **11–13** в таблице — **`forbidden`** на запуск вложенных агентов.

## Scope

### In Scope

- Роли **02–09** и **11–13**: назначение, типичные выходы, потребители, запреты.
- Роль **01** как оркестратор (не нумерованная роль из **02–09** / **11–13**, но входит в обзор **OR-001** через промпт оркестратора и матрицу артефактов).
- Связка **12 → 13** для записи в **`context/**`**.
- Разделение: код продукта vs канон; **13** не пишет product code.

### Out Of Scope

- Побайтовые JSON Schema каждой роли (держатся в файлах **`.cursor/agents/*.md`** — не дублировать здесь полностью).
- Роли **101–108** (подробно) — контур **100**; в глоссарии есть базовые определения.

## Current Reality Summary

- У ролей **02–09** ожидается **JSON-first** ответ (начало — машиночитаемый блок) и согласованный markdown-артефакт; **единого универсального envelope** для всех ролей нет — набор полей JSON зависит от роли и описан в её промпте.
- **01** агрегирует ожидания на переходах gate и ведёт таблицу producer/consumer ключевых файлов (постановка, ревью, план артефактов, код, отчёты **08**, ревью **09**).
- **13** обновляет **`context/**`** только после **12** и только в согласованных ветвях процесса.

## Examples

### Example 1: Happy Path — handoff **06 → 07 → 08**

**06** публикует план артефактов в каталоге сессии. **07** ревьюит и либо возвращает **06** на доработку, либо разрешает переход к **08**. **08** реализует только утверждённые задачи; **01** не перескакивает **07** без `stage_status`, разрешающего продолжение.

### Example 2: Partial — нужен канон, но команда ещё в **01**

Команда обсуждает в чате **01** формулировки будущего алгоритма без входа **`start-research`**. Это **не** даёт проверяемого пакета **101–108** и **не** создаёт SoT в **`context/algorithms/`**. Частичный выход: зафиксировать gap и перевести постановку на **`start-research`** с **100**, иначе **02** не может опереться на утверждённый канон.

### Example 3: Failure / Blocked — **13** без **12** или правки кода

Исполнитель просит **13** «сразу обновить канон» без **12**. По норме репозитория **13** опирается на инвентаризацию **12**; обход фиксируется как нарушение процесса. **13** **не** вносит правки в модули продукта вместо **08**.

## Inputs

| Вход | Описание |
|------|----------|
| Выход предыдущей роли | Согласованный **J** + **P**; **01** решает, достаточно ли для gate. |
| Канон SoT | Для тем под **`context/algorithms/`** — утверждённый файл для **02** и далее. |

## Outputs

Каждая роль производит **обязательный** машиночитаемый статус и согласованный текст/файлы; точные поля — в промпте роли; **универсального формата для всех ролей нет**.

## State Lifecycle

Артефакты живут в дереве сессии под управлением **01**; долговременный канон после утверждения — **`context/algorithms/`** и выбранные файлы **`context/**`** после **12–13**. Путаница **`plan/`** vs план **06** — см. [`glossary.md`](glossary.md).

## Commands

Проверки репозитория (**pytest**, flake8) — зона **11** и файл **`observability-and-tests.md`**; роли **02–07** не заменяют **11** запуском тестов «для галочки» без постановки.

## Observability

Доказательства перехода — **JSON `stage_status`**, отчёты и **status** у **01**. Роли **02–09** и **11–13** не добавляют отдельный обязательный централизованный event bus в этом каноне.

## Failure And Retry Rules

### FR-RA-1 — Нет записи в **`context/**`** мимо **13** (в типичном feature/fix)

После изменений кода обновление канона — через **12 → 13**, кроме явно других процессов, зафиксированных репозиторием.

### FR-RA-2 — **09 passed** не удаляет обязательность **11**

Ревью кода и независимая проверка остаются разными гейтами (см. топологию).

### FR-RA-3 — **Subagents: forbidden** для **02–09** и **11–13**

Повтор: роли **02–09** и **11–13** **не** вызывают Cursor Subagents для исполнения других ролей. Нарушение — блокер **OR-002**.

## Do Not Implement This As

- **Не** копировать полные JSON Schema из промптов — дублирование устареет молча.
- **Не** добавлять «опционально» для **12** перед **13** без явного исключения в другом процессе репозитория.
- **Не** оформлять исследование для канона как «ещё одну роль у **01**» вместо входа **`start-research`** и контура **100–108**.

## How start-feature / start-fix Must Use This

- **`02_analyst`** сопоставляет постановку с колонками **06**/**08**/**11** до фиксации ТЗ.
- **`06_planner`** трассирует задачи к ролям-потребителям и артефактам **P**; не смешивает каталоги плана (см. глоссарий).
- **`08_developer`** использует строку **08** как границу: код и отчёт **08**, не канон **`context/**`**.
- **`11_test_runner`** верифицирует, что цепочка артефактов не объявляет успех без фактических проверок.
- **`13_tech_writer`** обновляет этот файл при изменении владения артефактами.

## Технический контракт: handoff и запреты

| ID | Правило |
|----|---------|
| **H1** | Каждая роль **02–09** и **11–13**: первый блок ответа — **JSON-first** (`stage_status` и поля роли). |
