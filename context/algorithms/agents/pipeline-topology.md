# Топология мультиагентного пайплайна (Pipeline topology)

> **Аннотация:** маршруты **feature**, **fix** и **learn**, волны разработки (**08–09–11**), завершение через **11–12–13**, исследование для канона **только** через **`start-research`** (**100**, **101–108**), контракт **launcher** (кто вызывает вложенных субагентов) и сводная таблица терминальных состояний. Раздел дополняет [`glossary.md`](glossary.md) графом процесса.

## Status

`draft`

## Исходная цель (в пересказе)

Закрепить проверяемое описание того, **как** оркестратор ведёт сессию по ролям **02–09** и **11–13**: где ветвления, какие волны повторяются, чем заканчивается цикл и как это связано с точками входа **start-feature** / **start-fix** / **start-learn-project** и с **`start-research`** для канона.

## Why This Exists

Оператору и ролям **02** / **06** / **11** нужен общий язык про **порядок шагов** и **гейты** без чтения длинного промпта оркестратора. Иначе постановки задач смешивают «прошёл тесты» с «закрыта сессия» и путают контур **01** с контуром **100**.

## Связь с исходной постановкой

**OR-001** просит обзор ролей **02–09** и **11–13** под **01** под входом **start-feature** / **start-fix**: границы этапов, handoff JSON и markdown, таблица гейтов. Этот файл даёт **топологию** (маршруты и терминальные состояния); деталь матрицы «кто что пишет» — в [`roles-and-artifacts.md`](roles-and-artifacts.md).

**OR-002** требует явной оркестрации: кто запускает вложенных агентов и какие запреты у ролей **02+**. Здесь зафиксирован контракт **launcher** и разделение **01** (feature/fix/learn) и **100** (target-doc **101–108**), согласованное с правилами репозитория.

## Scope

### In Scope

- Высокоуровневые маршруты **feature**, **fix**, **learn** и типичные волны **08 → 09 → 11**.
- Финальный хвост **11 → 12 → 13** после разработки.
- Явное правило: **исследование доноров / текущего репо и синтез целевого алгоритма** выполняются **только** в контуре **`start-research`** (**100**, **101–108**); отдельной роли «исследователь» у **01** в нормативном пайплайне **нет** (устаревшее имя **`10_researcher`** снято из **`.cursor/`** по плану внедрения **G17.3**).
- Контракт **Subagents**: запуск только у **01** и **100**; остальные роли возвращают артефакты.
- Сводная таблица терминальных состояний сессии (агрегат правил оркестратора для проверяемости «закрыто vs частично»).

### Out Of Scope

- Полная машина состояний оркестратора со всеми ветками и лимитами итераций — источник истины в промпте **`01_orchestrator`** и проектных overrides; здесь только нормативная схема.
- Донорные паттерны и сравнительная матрица — файл **`donor-comparison.md`** (отдельный unit).
- Детальные JSON-схемы каждой роли — в **`roles-and-artifacts.md`**.

## Current Reality Summary

- **Feature:** после постановки (**02**) и проектных ролей типичная разработка идёт волнами **`08 → 09 → 11`**; могут быть дополнительные волны по решению **01**; затем финальный **11**, **12** (инвентаризация изменений), **13** (обновление согласованных фрагментов канона под **`context/**`**). Прогресс фиксируется **01** в статусе сессии и JSON-переходах; это **не** typed event-bus.
- **Fix:** ветвление после **02–03** на «короткий» или «полный» маршрут по согласованным правилам входа **start-fix**; чат оркестратора остаётся **01**, первый профильный шаг при достаточном входе — **02_analyst**.
- **Learn** (отдельное правило входа): ветка **`12 → 13`** без обязательного **08 / 09 / 11** для сценария обучения проекта.
- **Завершение** определяет **01** по файлу статуса, completion gate и overrides; успех финального **11** и прохождение **09** **не** подменяют друг друга и **не** равны автоматическому «completion» всей сессии.
- **Конфигурация `.cursor/`:** в актуальном дереве репозитория строка **`10_researcher`** и файл роли **не** используются; исследование для канона — только **`start-research`** (см. [`glossary.md`](glossary.md)).

## Target Behavior

Читатель открывает этот файл и понимает: (1) какой **путь** по ролям ожидается для feature, fix и learn; (2) что **исследование для канона** — только **`start-research`**, а не отдельная роль у **01**; (3) кто **единственный launcher** Subagents в контуре разработки (**01**), и что контур **100** отделён; (4) как интерпретировать **терминальное состояние** сессии.

## Target Flow

### TF1 — Вход и выбор контура

1. Пользователь выбирает правило входа: **start-feature**, **start-fix** или **start-learn-project** (или контур target-doc через **start-research** — см. пакет и будущий **user-story-handoff.md**).
2. **01** остаётся владельцем чата оркестрации в контуре feature/fix/learn; профильные роли **02–09** и **11–13** вызываются **только** **01** как Subagents (Cursor Subagents), не самим пользователем в обход **01**.

### TF2 — Feature: проектирование (типично **02–07**)

1. **02** получает intake: цель, ожидаемый результат, ограничения; при наличии утверждённого алгоритма под **`context/algorithms/…`** — этот файл — **SoT** для постановки.
2. Далее **03**, **04**, **05**, **06**, **07** по гейтам **01** (итерации ревью и планирования ограничены политикой оркестратора).

### TF3 — Feature: волны реализации (**08 → 09 → 11**)

1. **08** вносит изменения в код по утверждённому плану артефактов сессии.
2. **09** ревьюит код; результат **passed** у **09** **не** отменяет необходимость дальнейших шагов **01** и **не** равен завершению всего пайплайна.
3. **11** выполняет независимую проверку (типичный минимум — **pytest** и согласованные статические проверки из постановки).
4. **01** решает, нужна ли ещё одна волна **08 → 09 → 11** (например после доработок).

### TF4 — Завершение feature: **11 (финальный) → 12 → 13**

1. После последней необходимой волны — финальный **11** как gate качества.
2. **12** фиксирует инвентаризацию фактических изменений для канона.
3. **13** обновляет только согласованные области **`context/**`** по правилам репозитория; **13** **не** пишет код продукта.

### TF5 — Fix

1. Intake: наблюдаемое поведение, ожидаемое, воспроизведение, доказательства.
2. **02–03**; затем ветка **короткий** или **полный** маршрут по политике **01** и правилам **start-fix** (детали ветвления — в оркестраторе, не дублируются здесь дословно).

### TF6 — Learn (обзор репозитория)

1. Типовой маршрут learn обходится без обязательных **08 / 09 / 11** и ведёт к **12 → 13** по правилам входа **start-learn-project**.
2. **Терминальное состояние** сессии learn объявляет **01** по статусу и completion gate так же, как для других входов; успешный **13** **не** равен автоматическому **`completion`** без решения **01** (детали статусов и отчётности — в [`observability-and-tests.md`](observability-and-tests.md)).

### TF7 — Launcher и запрет смешения контуров

1. **Cursor Subagents** в этом репозитории для ролей implementation-пайплайна запускает только **01**; для target-doc — только **100**.
2. Роли **02–09** и **11–13** (и **04–07** явно в своих промптах) **не** инициируют вложенных агентов и **не** имитируют ответы других ролей.
3. Смешивать контур **100** и контур **01** в одной сессии без явной постановки пользователя **запрещено** нормой репозитория.

## Examples

### Example 1: Happy Path — одна волна разработки и закрытие

Пользователь запускает сценарий **start-feature**. **01** ведёт **02 → … → 07**, затем одну волну **08 → 09 → 11**. Все гейты выполнены. **01** запускает финальный **11**, затем **12** и **13**. Статус сессии отражает **completion** по правилам оркестратора; в канон попадают только согласованные правки **`context/**`** через **13**.

### Example 2: Partial / Recovery — нужна вторая волна

После первой волны **09** возвращает необходимость доработки. **01** инициирует вторую волну **08 → 09 → 11**. Частичное завершение волны **не** объявляется «ошибкой всего проекта» само по себе: решает **01** по статусу и лимитам итераций.

### Example 3: Failure / Blocked — неверная трактовка гейтов

Исполнитель считает, что «**11** прошёл» означает, что ревью **09** можно пропустить в следующей волне, или что **status=passed** у финального **11** автоматически закрывает сессию без **01**. Это **запрещённая** трактовка: **01** остаётся точкой решения о продолжении и завершении; **09** и **11** решают разные вопросы.

Краткая сводка: JSON **`stage_status`** у ролей, отчёты и статус сессии под **01**; отдельный обязательный stream-journal **нет**. Норматив по отчёту **11 → 01**, переменным **`AILIT_*`** в **`pytest`** и зазору **G1** — в [`observability-and-tests.md`](observability-and-tests.md).

## Failure And Retry Rules

### FR-PT-1 — Три независимых «успеха» не склеивать

**09 passed**, **11 passed** и **`completion` сессии** — разные утверждения. Повтор волны разрешён только пока **01** не исчерпал лимиты и не объявил блокировку / частичное завершение.

### FR-PT-2 — Запрет обхода launcher

Роль **02+** **не** запускает Subagents. Если в тексте роли появляется предложение делегировать через IDE другому агенту, это **нарушение** контракта **OR-002**.

### FR-PT-3 — Не подменять target-doc контуром **01**

Исследование для утверждения целевого алгоритма выполняется контуром **100** и ролями **101–108**, а **не** сессией **01** как замена **103**/**104**.

## Терминальные состояния сессии (сводная таблица)

Норма для читателя: состояние объявляет **01** и отражает в статусе сессии. Ниже — **агрегат** для проверяемости (точные имена полей — в промпте оркестратора).

| Категория | Смысл для человека | Типичные причины / следствия |
|-----------|---------------------|-------------------------------|
| **completion** | Сессия завершена успешно по правилам **01** | Выполнены гейты, нет обязательных open questions |
| **blocked** | Дальнейшие шаги невозможны без внешнего ввода | Blocker пользователя, отказ окружения, нарушение gate |
| **fix_by_research** | Нужен новый цикл **103 → волны → 104** по канону | Недостаточно фактов; не придумывать research в **104** |
| **fix_by_user** | Нужен ответ человека | Решение по scope, секретам, продуктовой политике |
| **partial / iteration_cap** | Исчерпан лимит итераций ревью/волн | Зафиксировать что сделано и что осталось |
| **external_tooling** | Останов по инфраструктуре IDE/Subagent | Продолжение после восстановления среды |

Пустое пересечение: **нельзя** объявлять **completion**, если явный gate **01** (например open question к пользователю) не закрыт.

## Do Not Implement This As

- **Не** описывать один плоский список «все роли подряд» без волн и финального хвоста — теряется смысл **08–09–11**.
- **Не** писать, что **100** «часть того же пайплайна**, что **08**» — смешение контуров.
- **Не** подменять отсутствие stream-journalа произвольным новым обязательным форматом событий только текстом этого канона без изменения правил репозитория.

## How start-feature / start-fix Must Use This

- **`02_analyst`** читает этот файл перед **`technical_specification.md`**, если задача затрагивает маршрутизацию **01** или гейты волны.
- **`06_planner`** трассирует задачи к шагам **Target Flow** и к **Терминальным состояниям**; нарезка сроков — в **`plan/17-agents-pipeline-canon.md`** (**106**), не внутри этого файла как план работ.
- **`11_test_runner`** проверяет, что сценарии проверки не объявляются заменой **09** или завершением сессии; форма отчёта и связка с **01** — в [`observability-and-tests.md`](observability-and-tests.md).
- **`13_tech_writer`** обновляет файл только если меняется согласованная модель пайплайна.

## Технический контракт: Launcher (OR-002)

| Правило | Формулировка |
|---------|--------------|
| **allowed** | **`01_orchestrator`** запускает Subagents ролей **02–09** и **11–13** в контуре feature/fix/learn. |
| **allowed** | **`100_target_doc_orchestrator`** запускает Subagents ролей **101–108** в контуре target-doc. |
| **forbidden** | Роли **02–09** и **11–13** (и любые **не** перечисленные как **01**/**100**) — инициировать **Cursor Subagents** для исполнения других ролей. |
| **required** | Ответ роли остаётся **JSON-first** согласно глоссарию. |
