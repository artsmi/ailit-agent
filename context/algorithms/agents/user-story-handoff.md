# USER-story → start-research / start-feature / start-fix (Handoff contract)

> **Аннотация:** минимальный контракт постановки от человека: поля intake, дерево решений **без** магической NL-матрицы, связь с ролями **02+**. Текст самодостаточен для оператора и **02**.

## Status

`draft`

## Связь с исходной постановкой

**OR-006** требует описать handoff **USER-story** в **`start-research`** и **`start-feature`**: формат, точки решения и артефакты между этапами. **CR-F10** и **CR-F11** из свода фактов: универсальной таблицы «любая фраза → только research» **нет**; жёсткое правило — **только утверждение target-doc без кода** → **`start-research`** / **100**, не чат **01** как замена **103–104**. Утверждённый **`context/algorithms/**`** в implementation-контуре — **SoT**, это **не** создание канона.

## Why this exists

В большом репозитории одна неформулированная story порождает параллельные «истины»: кто-то пишет код по памяти, кто-то — markdown в чате. Контракт ниже переводит story в **режим** и **минимальный набор полей**, которые **01** и **02** могут проверить.

---

## Дерево решений (норматив)

### Слой для человека

1. **Нужен только утверждаемый алгоритм / канон под `context/algorithms/`, без изменения product-кода?**  
   → Да: открыть **`start-research`**, контур **100–108**. **Запрещено** поручать **103–104** из чата **01** без Subagent pipeline.

2. **Нужна поставка кода / исправление в продукте?**  
   → Да: **`start-feature`** или **`start-fix`** под **01**. Далее — **mode detection** (feature vs fix) по правилам входа.

3. **Сначала канон, потом код?**  
   → Два последовательных контура: **A)** **`start-research`** до user OK канона; **B)** затем **`start-feature`** с явной ссылкой **02** на утверждённый файл **`context/algorithms/...`**.

### Технический контракт (decision outcomes)

| Условие | Выходной маршрут | Owner чата / оркестрации |
|---------|------------------|---------------------------|
| Только target-doc / алгоритм | **`start-research.mdc`** | **100** + роли **101–108** |
| Изменение продукта (фича) | **`start-feature.mdc`** | **01** + **02–09** и **11–13** по топологии |
| Исправление дефекта | **`start-fix.mdc`** | **01** + **02–09** и **11–13** (ветка fix) |
| Обучение / learn-project без волн 08/09/11 | **`start-learn-project.mdc`** | **01**; хвост **12→13** без типичных волн разработки |

**Forbidden:** интерпретировать отсутствие ключевых слов в свободном тексте как разрешение пропустить **100**, если цель — утверждение канона.

## Минимальная схема полей intake

### Общие поля (feature / fix / hybrid после research)

| Поле | Обязательность | Смысл |
|------|----------------|-------|
| `user_goal` | **required**, non-empty string | Что человек хочет получить на выходе цикла. |
| `expected_result` | **required** | Измеримый или проверяемый результат (команда, артефакт, поведение). |
| `constraints` | **default** empty list | Сроки, риски, «не трогать X», политика live/test. |
| `repo_topic` | **required** для масштаба | Область репозитория или пакет (для монорепо). |
| `algorithm_sot_path` | **nullable** | Путь **`context/algorithms/...`**, если канон уже утверждён и должен стать SoT для **02**. |

### Поля специфично для **fix**

| Поле | Обязательность |
|------|----------------|
| `observed` | **required** |
| `expected` | **required** |
| `repro` | **required**, если дефект воспроизводим иначе «не проверить». |
| `evidence` | **default** empty; логи, скриншоты CLI — **без** секретов. |

### Поля для **target-doc only** (маршрут **100**)

| Поле | Обязательность |
|------|----------------|
| `target_topic` | **required** — тема канона (имя пакета или файла под `context/algorithms/`). |
| `scope_in` / `scope_out` | **required** на уровне «перечень из 1–5 пунктов»; **forbidden** оставлять пустыми «потом». |
| `non_code_intent` | **required** boolean true — явная фиксация «код продукта не меняем в этом контуре». |

## Связь с ролями **02+** (после входа **01**)

| Роль | Что потребляет из USER-story / intake |
|------|----------------------------------------|
| **02_analyst** | `user_goal`, `expected_result`, `constraints`, **`algorithm_sot_path`**; для fix — `observed` / `expected` / `repro`. |
| **03_tz_reviewer** | ТЗ от **02** + trace к канону, если заявлен SoT. |
| **04_architect** | Границы архитектуры из **02** и нефункциональные ограничения. |
| **06_planner** | План артефактов; не смешивает с **`plan/`** корня (см. **glossary**). |
| **08_developer** | Задачи из плана **06**, не «переписать канон». |
| **11_test_runner** | Команды из канона / ТЗ; статусы → **01**. |
| **12 → 13** | Инвентаризация и точечное обновление **`context/**`** при изменении устойчивой картины. |

Исследование для будущего канона (**доноры**, **текущий репо**, синтез, human approval) выполняется **только** через **`start-research`** и контур **100–108**; в сессии **01** эту работу **не** подменяют.

## Examples

### Example 1: Happy path — сначала research (target-doc only)

Пользователь формулирует: нужен утверждаемый канон пакета **`context/algorithms/agents/`**, код не менять. Intake: `non_code_intent=true`, `target_topic=agents-pipeline`, `scope_in` перечисляет роли **02–09** и **11–13** под **01** и USER-story, `scope_out` — product runtime.

**Система:** человек открывает **`start-research`**, **100** ведёт **101–108**; **01** **не** пишет `synthesis.md` вместо **103**.

**Ожидаемый результат:** черновик и затем утверждаемый канон в **`context/algorithms/...`**, human approval package, план **`plan/<NN>-*.md`** от **106**.

---

### Example 2: Partial / hybrid — канон уже есть, нужна фича

Пользователь ссылается на **`context/algorithms/agent-memory/INDEX.md`** как на SoT и просит изменить продукт в соответствии с §Target Flow.

Intake: `algorithm_sot_path` заполнен, `user_goal` описывает изменение кода, `constraints` запрещает менять канон без отдельного target-doc.

**Система:** **`start-feature`**, **02** цитирует канон в ТЗ, **06** трассирует задачи к шагам канона.

**Partial допустим:** если при разработке обнаружили расхождение кода и канона — **blocked** по процессу: либо **start-research** на обновление канона, либо явное решение человека (не «тихая» правка).

---

### Example 3: Failure / blocked — target-doc в чате **01**

Пользователь просит «синтезируй synthesis и напиши канон здесь» в сессии **01** без **100/103**.

**Система:** процесс **blocked** относительно **OR-006** и **start-research** протокола: owner-артефакты **103/104** не вызываются.

**Выход:** перенаправление на **`start-research`**; **100** фиксирует полный запрос по правилу входа target-doc, без поручения **103–104** из чата **01**.

## Failure and retry rules

### FR-USER-1: Неполный intake

Если отсутствует **`expected_result`** или **`user_goal`**, **01** **forbidden** запускать **02** до дополнения (iterative intake человеком).

### FR-USER-2: Ложный hybrid

Если заявлены и «только док», и «срочно код», **required** явное решение человека: порядок контуров **A затем B** или waiver с риском (оформляется вне этого файла в **open_questions** target-doc).

## How start-feature / start-fix must use this

- **`02_analyst`** **required** копирует в ТЗ минимальные поля intake и ссылку на **`algorithm_sot_path`**, если не **null`.
- **`06_planner`** поддерживает трассировку к **`context/algorithms/...`** вместо дублирования канона.
- **`01_orchestrator`** не подменяет **100** при **`non_code_intent=true`**.

## Acceptance criteria

- Есть дерево решений с исходами **research / feature / fix / learn**.
- Есть таблица полей с **required / default / nullable / forbidden**.
- Три примера: **happy research**, **hybrid feature+алгоритм**, **blocked** нарушение протокола.
