# PAG: стабильная индексация и независимость от запроса

## Инвариант запроса

- **Запись в PAG** (индексатор `PagIndexer`, W14-команды `summarize_c` / `summarize_b` → `SqlitePagStore`) **не зависит** от текущего пользовательского `goal` / `user_subgoal`. Суммаризации для узлов C и B описывают **нейтральное** содержимое репозитория (чанк / агрегат детей), а не формулировку запроса.
- **Выбор путей** и планерский раунд могут использовать `goal` для маршрутизации и ответа; это **не** смешивается с fingerprint узлов PAG и не должно подмешиваться в сохранённые `summary` / `attrs` при записи через W14 write path.

## Производные поля (защита при reindex)

Поля, которые считаются **производными** от LLM или downstream W14 и **сохраняются** при инкрементальном sync, если идентичность C-узла и хэш содержимого сегмента не изменились:

- на **C:** `summary` (поле узла), `attrs.summary_fingerprint`, все `attrs` с префиксом `w14_`;
- на **B (file):** `summary` (если уже есть валидная B-суммаризация), `attrs.summary_fingerprint`, `attrs.child_basis_fingerprint`, `attrs.w14_*`;
- на **A (project):** при обновлении метаданных репозитория выполняется merge `attrs`: новые поля контекста перекрывают старые, прочие ключи из старого слоя сохраняются.

## Индексация C

Инкрементальный sync по файлу **не** удаляет все C по path и не пересоздаёт граф с нуля: удаляются только узлы, исчезнувшие из декомпозиции; для совпадающего стабильного `node_id` при неизменном `content_fingerprint` обновляются структурные атрибуты (строки, title и т.д.), производные поля переносятся.

## Отсечка B (W14 runtime)

Повторный вызов LLM для `summarize_b` не выполняется, если сохранённый `child_basis_fingerprint` совпадает с базисом, вычисленным по текущему набору дочерних C, переданных в команду, и у B есть валидный сохранённый summary.
