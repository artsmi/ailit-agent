# DeepSeek Live Integration Test Contract (Cross-cutting)

## Назначение

Зафиксировать единый контракт **реальных** интеграционных тестов через DeepSeek на протяжении этапов `3+`.

Этот документ описывает политику секретов, минимальные сценарии и gate для разработки `ailit-agent`.

## Политика секретов (обязательная)

- API ключ **никогда** не хранится в git: ни в markdown, ни в коде, ни в fixtures.
- Локально и в CI ключ передается только через secret mechanism:
  - переменная окружения `DEEPSEEK_API_KEY`;
  - опционально `AILIT_LIVE_PROVIDER=deepseek` для явного включения live режима.
  - опционально локальный файл `config/test.local.yaml` (в git не коммитится; шаблон — `config/test.local.yaml.example`): при отсутствии `DEEPSEEK_API_KEY` допускается чтение `deepseek.api_key` только из этого файла.
- Любой ключ, который оказался в чате/логе/репозитории, считается скомпрометированным: **rotate/revoke** в консоли провайдера.

## Что именно является "контрактом"

Для каждого нового этапа, где появляется исполнение модели, команда добавляет или расширяет live тесты так, чтобы:

1. **Small smoke** оставался быстрым и дешевым.
2. **Large scenario** покрывал сквозной realistic trace (tools/workflow/session), но запускался реже.

Оба типа тестов используют DeepSeek как **единый live baseline provider** до появления отдельной политики multi-provider live matrix.

## Feature flag запуска

Live тесты должны быть выключаемыми по умолчанию:

- если `DEEPSEEK_API_KEY` отсутствует, live тесты пропускаются (`skip`), а не падают;
- если ключ есть, live тесты выполняются.

Дополнительно рекомендуется guard:

- `AILIT_RUN_LIVE=1` обязателен для suite, которая делает сетевые вызовы в CI (чтобы случайно не жечь квоту).

## Small smoke contract (обязательный для каждого этапа с моделью)

Цель: подтвердить, что интеграция живая и нормализация ответа работает.

Минимальные требования:

- один сетевой вызов к DeepSeek chat completions (или актуальному endpoint адаптера);
- строгие проверки:
  - HTTP успех;
  - непустой `content` **или** корректный tool payload (если этап про tools);
  - наличие нормализованного usage поля, если endpoint возвращает usage (если нет — явный `usage_missing=true` в нормализованном результате, но это должно быть зафиксировано как известное ограничение).

Ограничения:

- маленький `max_tokens`;
- короткий system/user prompt;
- без записи на диск, без больших вложений.

## Large scenario contract (обязательный как периодический gate)

Цель: подтвердить, что `ailit-agent` держит реальную "работу" end-to-end.

Минимальные требования:

- минимум 3 осмысленных "шага" на уровне продукта (например: прочитать вход → предложить план → выдать структурированный результат);
- если на этапе уже есть tools: минимум один успешный tool roundtrip;
- если на этапе уже есть workflow layer: события должны включать переход stage (по контракту `stage.entered` / `task.*`).

Нефункциональные требования:

- бюджет: лимит токенов и wall-clock timeout на уровне теста;
- стабильность: ретраи rate limit только в transport layer, тест фиксирует финальный статус `ok/degraded` явно.

## Артефакты, которые live тест обязан сохранить локально (когда runtime уже пишет `.ailit/`)

Для large scenario:

- `events.jsonl` содержит достаточно событий, чтобы восстановить прохождение сценария;
- `state.json` отражает финальный статус;
- `telemetry/usage.jsonl` не пустой (если провайдер отдает usage).

## Связь с observability этапа 2

Live тесты должны проверять не "красивый stdout", а наличие данных для UI map:

- минимальный набор `event_type` присутствует;
- correlation поля заполнены хотя бы на уровне `run_id` + `session_id` + `workflow_id` (когда доступно).
