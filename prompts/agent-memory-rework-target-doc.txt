## Why Rework Is Needed

Текущий результат полезен как навигационный канон, но недостаточно полный и прозрачный для человека.

Главные замеченные слабости:

1. `context/algorithms/agent-memory/prompts.md` слишком тонкий: он не раскрывает prompts AgentMemory для всех состояний, хотя это было требованием исходной постановки.
2. Не хватает human approval package:
   - что пользователь утверждает;
   - что не утверждается;
   - где разделы thin/partial;
   - какие gaps и TBD остаются;
   - какие waivers нужны или запрещены.
3. Не хватает строгой source request coverage matrix:
   - каждый блок исходного запроса → раздел документа → глубина покрытия → gaps.
4. Не хватает target doc quality matrix:
   - human clarity;
   - technical completeness;
   - examples;
   - commands/proof;
   - gaps;
   - verdict.
5. Некоторые разделы смешивают current reality, target behavior и implementation backlog; это нужно сделать более прозрачным для человека.
6. Некоторые target/TBD элементы допустимы, но должны быть вынесены в явный `open_gaps_and_waivers.md`, а не теряться в тексте.

### 2. Prompts Section Must Be Expanded

`context/algorithms/agent-memory/prompts.md` должен быть существенно усилен.

Минимально он должен описывать:

- system prompt для AgentMemory;
- planner / `plan_traversal` prompt;
- `summarize_b` prompt;
- `summarize_c` prompt;
- `finish_decision` prompt;
- `propose_links` target prompt;
- repair / invalid response prompt;
- prompts для multi-language и documentation files;
- forbidden outputs;
- examples good/bad;
- invalid output + recovery examples;
- what runtime may show to LLM;
- what runtime must never show to LLM;
- schema-like input/output contract for every prompt role.

Если для части prompts current code не содержит отдельной константы, это нужно явно пометить как:

- `Current reality`: где это сейчас находится или отсутствует;
- `Target behavior`: какой prompt contract должен появиться;
- `Gap / start-feature implication`: что должен реализовать будущий feature.

### 3. Source Request Coverage

`source_request_coverage.md` должен покрыть исходный запрос из `context/artifacts/target_doc/original_user_request.md`.

Таблица должна иметь минимум:

| Original request block | Covered in | Depth | Missing | Needs user decision |

Depth:

- `full`
- `partial`
- `thin`
- `missing`

Если раздел `prompts` остаётся `thin`, approval запрещён без явного user waiver. Для этого конкретного rework user waiver на `prompts.md` **нежелателен**: ожидание пользователя — получить полный и прозрачный prompt contract.

### 4. Quality Matrix

`target_doc_quality_matrix.md` должен оценить каждый файл:

- `INDEX.md`
- `runtime-flow.md`
- `memory-graph-links.md`
- `llm-commands.md`
- `prompts.md`
- `external-protocol.md`
- `failure-retry-observability.md`

Колонки:

| Section/File | Human clarity | Technical completeness | Examples | Commands / proof | Gaps | Verdict |

Если core section имеет `low`, `25` не должен давать `approved_for_user_review`.

### 5. Gaps And Waivers

`open_gaps_and_waivers.md` должен явно перечислить:

- implementation backlog;
- target event names TBD;
- current-vs-target gaps;
- sections that need future `start-feature`;
- waivers, если предлагаются.

Waiver запрещён для:

- отсутствующего Source User Goal;
- отсутствующего Target Flow;
- отсутствующих examples для workflow;
- отсутствующих Acceptance Criteria;
- отсутствующего prompts contract по исходному требованию;
- отсутствующего provenance.

### 6. Canonical Docs Must Stay Discoverable

Если target docs обновляются, сохранить layout:

- `context/algorithms/agent-memory/INDEX.md` как главный вход;
- ссылка из `context/algorithms/INDEX.md`;
- понятный "How to read this document";
- "How start-feature / start-fix must use this";
- glossary / terms, если нужно.
