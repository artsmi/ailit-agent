Работай строго по актуальному плану и правилам workflow.
Цель: выполнить этапы:

G14R.11 — Интеграция, README/context и удаление старого planner

Порядок обязателен: сверху вниз. Нельзя переходить к следующему этапу, пока текущий не выполнен, не проверен и не закоммичен.
Перед началом:
1. Полностью прочитай `plan/14-agent-memory-runtime.md`.
2. Полностью прочитай `.cursor/rules/project-workflow.mdc`.
3. Найди в плане:
  - задачи этапа;
  - критерии приёмки;
  - проверки;
  - обязательные контракты/схемы/ссылки;
  - зависимости от предыдущих этапов;
  - файлы/anchors, которые нужно трогать или проверить.
4. Кратко выпиши, какие пункты конкретно будешь закрывать каждого этапа.
Для каждого этапа отдельно:
1. Реализуй только scope текущего этапа.
2. Не закрывай этап “примерной” реализацией: каждый пункт задач и критериев приёмки должен быть покрыт кодом, тестом или явным объяснением, почему пункт уже был реализован ранее.
3. Если в плане есть schema/contract/event/payload — реализация должна соответствовать им буквально.
4. Если есть существующий runtime path, исправляй/интегрируй его, не создавай параллельный обходной модуль.
5. Добавь/обнови тесты, указанные в плане. Если тестов в плане недостаточно, добавь минимальные regression tests на контракт этапа.
6. Запусти проверки из плана. Минимум:
  - `pytest` по затронутым Python tests/packages;
  - `flake8` по затронутым Python-файлам;
  - если затронут desktop — `npm test` / `npm run typecheck` в `desktop/`.
7. Если проверки не проходят — исправь, повтори проверки.
8. После успешных проверок сделай отдельный коммит с префиксом этапа.
9. После коммита отправь ntfy-уведомление по правилу `.cursor/rules/project-workflow.mdc`.
10. Только после этого переходи к следующему этапу.
После выполнения этапов:
1. Полностью перечитай критерии закрытия workflow.
2. Проверь README/context: если по правилам workflow статус должен быть обновлён — обнови.
3. Если workflow действительно закрыт:
  - обнови README/status;
  - сделай отдельный коммит закрытия workflow;
  - отправь ntfy.
4. Если workflow не закрыт:
  - НЕ закрывай его формально;
  - дай список оставшихся пунктов: этап, пункт плана, файл/контракт, что именно не сделано.
5. В финальном ответе укажи:
  - какие этапы закрыты;
  - какие проверки запущены;
  - какие коммиты созданы;
  - есть ли оставшийся scope.
Запреты:
- Не делай один общий коммит на оба этапа.
- Не пропускай проверки.
- Не закрывай workflow, если хотя бы один acceptance criterion не подтверждён.
- Не заменяй contract implementation моками.
- Не создавай новую параллельную реализацию, если в плане указан существующий path/anchor.
- Не меняй unrelated files.
