# Workflow: рецепт токен-экономии и shared KB (ветка планирования)

Документ фиксирует **цель**, **выбранный архитектурный паттерн** и **минимальные канонические ограничения** для следующей итерации реализации. Детальная разбивка на этапы и задачи — в [`workflow-hybrid-memory-mcp.md`](workflow-hybrid-memory-mcp.md).

## Цель

Снизить расход контекста за счёт **вынесения устойчивых знаний** из «простыни» чата в **внешний слой памяти** с **детерминированным извлечением** (MCP / KB), не ломая локальную рабочую память агента.

## Выбранный паттерн: hybrid memory

- **Приватная рабочая память** (сессия, scratch, черновики) остаётся в runtime агента и не подменяется «сырой» выгрузкой логов в KB.
- **Общий слой (shared / KB)** хранит нормализованные факты, решения, инварианты, ссылки на артефакты — с явными **областями видимости** и политикой записи.

Подтверждение выбора и дальнейшие шаги — в связанном workflow [`workflow-hybrid-memory-mcp.md`](workflow-hybrid-memory-mcp.md).

## Канонические ограничения для следующей итерации (без расширения scope)

1. **Области (scopes):** любая запись в shared-слой маркируется как минимум одним из уровней: `org` | `workspace` | `project` | `agent` | `run`. Значения по умолчанию и наследование описываются только в `workflow-hybrid-memory-mcp.md`, не «в коде на глаз».
2. **Запрет сырого дампа:** полные логи чата, stack traces и неструктурированные простыни **не** пишутся в KB без нормализации (выжимка, класс сущности, ссылка на файл/коммит).
3. **Index-first retrieval:** перед подмешиванием большого объёма в промпт обязателен шаг **поиска/ранжирования** (ключевые слова, граф, вектор — конкретика в этапах hybrid workflow), а не «прочитать весь vault».
4. **Конфликты и версии:** при противоречии двух записей приоритет правил разрешения фиксируется в hybrid workflow (время, `supersedes`, ручное разрешение) — до реализации согласовать с постановкой.
