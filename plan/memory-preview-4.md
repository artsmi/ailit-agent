3. «Промпт с kb_» не годится — как у доноров и что значит «гарантия в рантайме»
Честно:
В текущем ailit (по коду) долгая память в SQLite завязана на явные kb_* tool calls от модели. Автоматического фона вида «каждый turn без участия LLM пиши сущности в граф/вольт» на уровне SessionRunner в той же форме, что у MCP-доноров (Obsidian, Graphiti) (отдельный сервер, свои схемы write), здесь не дублируется в одной фразе «уже в коде сессии гарантируется» — иначе вы бы не видели нулевой KB при чисто read_file/run_shell.
У доноров (Letta, отдельные memory server): типичны (а) плановые write-path (сводка, episodic, sync), (б) отдельные трассы, не «только то, что модель пожалела вызвать вручную».
Что можно иметь в виду продуктово (без обещания, что оно уже так в вашей ветке): post-turn суммаризатор → запись в KB, второй агент, подсистема memory на хуках после model.response — это уже дизайн-изменения, а не «пользовательский промпт». В Ask mode я не реализую, но направление такое.

Прямого аналога «суммировать 711 * restored file slots» в Graphiti нет — у них другая метрика: эпизоды, граф, retrieval hit rate и т.д. Сходство по духу одно: окно контекста конечно → нужно либо обрезать, либо вынести знания наружу, либо дотянуть критичные куски ограниченно. У вас третье — узкоспециальное (недавние read_file), у доноров чаще второе + retrieval.
Claude Code / coding agents ближе по духу: кэш «недавно смотрел», резюме длинного чата, re-read политика — но реализация везде своя; это не «стандарт индустрии один в один».

Ниже — сопоставление по смыслу (часть — из ваших локальных реп claude-code / opencode, часть — общая практика «доноров» и то, что уже есть у вас в ailit).
Claude Code (по вашему зеркалу репо)
Ограничение «длинного цикла» там не одной кнопкой, а пакетом:
Права на инструменты — в README явно: не просто allow/deny, а классификация риска (LOW/MEDIUM/HIGH), автоматические решения, отдельная подсистема в tools/permissions/. Это режет опасные и бессмысленные действия до выполнения (в т.ч. часть «бесконечного bash»).
Бюджет на результаты тулов — в toolResultStorage.ts: агрегатный лимит на размер результатов в сообщении, персист больших кусков на диск с превью в контексте, стабильные решения «что заменить», чтобы не раздувать промпт и кэш. Это не лимит числа вызовов, но сильно снижает цену каждого хода и тормозит сценарий «каждый раз тащим мегабайты в чат».
Скиллы — в README: SkillTool как отдельный инструмент для пользовательских скиллов (описание «когда вызывать» — часть продукта скилла, не «промпт пользователя про kb_»).
Субагенты / итерации — в коде есть явные итерации сварма/подзадач (inProcessRunner и т.п.); для удалённых сессий в ccrSession фигурируют в т.ч. error_max_turns — т.е. верхняя граница по ходам/фазам — на уровне продукта, не «модель сама остановится».
Итого по духу: жёсткие границы (ходы/фазы) + права + экономия контекста на тул-выводе + скиллы как структурированные процедуры, а не только «системный текст».
OpenCode (по сгенерированным типам SDK в вашем opencode)
В AgentConfig прямо задокументировано:
steps / maxSteps — «Maximum number of agentic iterations before forcing text-only response» — т.е. после N шагов агентного цикла продукт принудительно уводит модель в только текст (обрывает цепочку tool-calls как стратегию продолжения).
permission — по типам: отдельные правила на bash, task, doom_loop, skill, и т.д. — т.е. детект/ограничение «плохих» паттернов цикла (doom loop) — отдельная ось, не только «макс. шаги».
У агентов есть description / prompt — когда какой агент/режим использовать (это уже продуктовый слой, ближе к «скиллу» или к ролям).
Итого: жёсткий cap итераций агента + permissions, включая anti–doom-loop + роли/промпты агентов.
Что делают «доноры» с множеством вызовов тулов
Обычно комбинация, а не одно:
Механизм	Эффект
Лимит шагов (OpenCode steps)	После N итераций — стоп tool-only, нужен финальный ответ или стоп сессии
Права / классификация риска (Claude Code)	Меньше бессмысленных/опасных bash и записей
Бюджет вывода + превью/файл (Claude Code, у вас — prune/pager/budget)	Много вызовов не раздувает контекст так же быстро
Субагенты с отдельным бюджетом	«Длинное» — в узком контексте, основной цикл короче
Внешняя память + retrieval (Graphiti, vault MCP)	Меньше нужды сканировать репо сотнями read_file
Как «сделать так же» у вас (ailit) — без реализации, только направления
У вас уже есть часть слоя M3: max_turns, tool output budget / prune, pager, compaction + post-compact restore, tool exposure. Проблема вашего примера — модель укладывается в правила, но легально делает очень много «лёгких» шагов (read_file/list_dir/run_shell).
Чтобы приблизиться к Claude Code / OpenCode:
Явный «agent steps» cap помимо max_turns — например, считать только ходы, где ответ модели = tool_calls, и после K таких подряд (или за сессию) принудительно требовать текстовый ответ без тулов или останавливать с понятной причиной (аналог OpenCode steps).
Doom-loop / паттерн-детектор — эвристика: одни и те же тулы с теми же аргументами, рост без прогресса, «снова list_dir корня» — предупреждение или deny (аналог doom_loop в OpenCode + идея permission слоя).
Ужесточение прав на run_shell для «исследовательских» задач — таймауты, запрет интерактивных лаунчеров, allowlist команд (у вас уже есть approval/permission engine — можно политикой сузить сценарий как у доноров).
Скиллы / сценарии продукта — не просить пользователя знать kb_*, а встроенный режим «исследование репо → записать в KB» как отдельный workflow или системный блок + ограниченный набор тулов на шаг (как роль subagent в OpenCode / режим в Claude Code).
Фоновая запись в KB (отдельная тема) — тогда «память» не зависит от того, выбрала ли модель kb_write_fact в этот тик.
Промпты и скиллы у доноров
Claude Code: да — системные описания тулов, SkillTool для пользовательских процедур, README описывает permission и кэш схем.
OpenCode: да — prompt / description у агента, permission по типам инструментов и skill, плюс steps в конфиге агента (см. типы выше).
То есть «описание промптов» у них — часть конфигурации продукта (агент, скилл, политика), а не обязанность конечного пользователя знать внутренние имена kb_*.
Коротко: у OpenCode «длинный цикл» режется в первую очередь steps → принудительно text-only и doom_loop в permissions; у Claude Code — сложные permissions, бюджет/замена тул-результатов, скиллы, лимиты/ошибки по max turns в отдельных режимах. Вам «сделать так же» — добавить продуктовые аналоги steps + anti-doom и при желании режим/скилл под «память репо» без раскрытия имён тулов пользователю; технически это уже постановка на доработку поверх текущего SessionRunner и UI.
