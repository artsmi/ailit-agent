Коротко, почему растёт число D
D после компакта — это не «одна сводка на весь чат», а отдельная запись в граф памяти на каждый раз, когда сессия решает: «история уже длинная, старое выкидываем из промпта, но сохраняем краткое резюме в PAG».

Так происходит потому что:

Условие срабатывания частое: как только в полной истории больше сообщений, чем хвост (по умолчанию 40), при включённой опции записи в память перед каждым следующим шагом к модели снова выполняется компакт.
Новая D каждый раз: идентификатор компакта строится с новым UUID, прошлые D не переиспользуются — поэтому длинный диалог с многими turn’ами превращается в цепочку из многих узлов compact_summary, а не в одну «накопительную» ноду.
Итог человеческим языком: каждый «срез» длинной переписки при очередном обращении к модели оставляет свой след в графе, поэтому D размножаются быстрее, чем кажется по смыслу «мы же просто сжали контекст».

Сжатие у вас не через отдельный запрос к LLM для «умного» пересказа. Оно делается правилами в коде.

1. Обрезка под промпт (compact_messages)
Файл tools/agent_core/session/compaction.py: из полной истории остаётся хвост из tail_max сообщений (в настройках сессии, по умолчанию 40), плюс у длинных tool-ответов текст обрезается по лимиту символов и дописывается ...[truncated]. Это чистая механика, без модели.

2. «Компакт в память» (DLevelCompactService)
Файл tools/agent_core/session/d_level_compact.py: для части истории, которая выпала из хвоста, строится короткий текст методом _summarize: берутся до 8 сообщений с конца удалённого куска, роли и текст сжимаются пробелами, каждое поле режется до 240 символов, строки склеиваются в фиксированный шаблон вроде «D-level compact summary.» + буллеты. Снова без LLM — детерминированная сборка строк.

Итог: LLM для сжатия контекста здесь не вызывается; модель по-прежнему вызывается по обычному циклу сессии, а «компакт» — это усечённый список сообщений + локально сгенерированное system-сообщение с резюме и запись узла в PAG.
