## 1. Итоговая цель

1. **Любой репозиторий и любой каталог:** после установки пользователь запускает `ailit` из произвольной точки ФС, указывает или сменяет «текущий проект» (корень git/рабочий каталог), переключается между репозиториями без переустановки и без привязки shim к одному клону `ailit-agent` как к единственному смыслу «дома».
2. **Глобальная конфигурация:** каноническое дерево пользователя (аналог `~/.claude` и XDG-слоёв) хранит провайдер, ключи (через env / secure storage), дефолтные модели, лимиты; проектные файлы (например `.ailit/config.yaml` вверх по дереву) переопределяют глобальное предсказуемо.
3. **`ailit agent` по умолчанию = полноценный TUI:** интерактивная среда (бывший `ailit tui`): slash-команды, настройка глобального слоя из TUI, работа с DeepSeek и другими провайдерами наравне с chat. **`ailit agent run`** остаётся отдельным режимом для изолированного/сценарного запуска (CI, YAML, «чистый» контекст).
4. **Потоковый UX:** и `ailit chat`, и интерактивный `ailit agent` показывают ход рассуждения и токены ответа **по мере поступления**, без единого огромного блока в конце длинного запроса.
5. **Скрытый промптинг:** явная «карта» системных фрагментов (базовый агент, окружение, инструменты, политики compaction/shortlist), версионирование и минимизация дубликатов в духе OpenCode (модули `.txt`) и приоритетов Claude Code (`buildEffectiveSystemPrompt`), с **оптимизацией токенов** как сквозной фичей `ailit` (см. идеи удержания контекста в context-mode).
6. **Деплой одной командой:** из корня клона — «продакшен»-установка (не обязательно editable); обновление и удаление **не стирают** глобальные пользовательские настройки без явной команды «factory reset».

Критерий успеха продукта: разработчик один раз ставит CLI, настраивает глобально из TUI или CLI, открывает любой проект, работает агентом в терминале с приятным потоковым UI, при обновлении пакета не теряет `~/.ailit` (или XDG-эквивалент).

## 2. Диагностика текущего состояния (точки роста)

### 2.1. Установка и shim указывают на клон

Скрипт `scripts/install` создаёт симлинк на **venv текущего клона**; последний успешный install «перезаписывает» shim — это удобно для разработки, но не задаёт модель «глобальное приложение + обновляемый бинарник/venv в фиксированном месте».

```28:36:/home/artem/reps/ailit-agent/scripts/install
LOCAL_BIN="${HOME}/.local/bin"
...
    ln -sf "${VENV}/bin/ailit" "${LOCAL_BIN}/ailit"
    echo "Команда ailit:  ${LOCAL_BIN}/ailit  →  ${VENV}/bin/ailit"
```

**Вывод:** нужна команда/режим **production install** (целевой префикс, отдельный venv или wheel, политика обновления), не ломающая глобальный конфиг.

### 2.2. TUI: служебная разметка Rich и синхронный вывод хода

В лог чата пишется разметка вида `[dim]user>[/dim]`, что даёт пользователю «сырой» вид (`DIM`, теги) вместо чистого оформления.

```150:165:/home/artem/reps/ailit-agent/tools/ailit/tui_app.py
        log.write(f"[dim]user>[/dim] {line}")
...
            text, st, usage = chat.run_user_turn(
                line,
                state=self._app_state.session_view(),
                diag_sink=handle.sink,
                log_path=str(handle.path),
            )
...
            log.write(escape(text))
```

**Вывод:** визуальные роли «пользователь / ассистент / система» должны задаваться стилями Textual/Rich без утечки markup в пользовательский текст; вывод ассистента — **инкрементально** (см. этап DP-3), а не одним `log.write(text)` после полного `run_user_turn`.

### 2.3. Session loop уже умеет stream-режим на уровне настроек

В ядре зафлажен потоковый режим — нужно протащить его в UI и агрегировать дельты.

```64:78:/home/artem/reps/ailit-agent/tools/agent_core/session/loop.py
@dataclass(frozen=True, slots=True)
class SessionSettings:
    """Параметры одного прогона session loop."""
...
    use_stream: bool = False
```

**Вывод:** DP-3 — в основном wiring: провайдер → события → chat/TUI, с тестами на уменьшение «блочности».

### 2.4. CLI: `tui` отдельно от `agent`

Сейчас TUI висит на подкоманде `tui`, а сценарный запуск — на `agent run` ([`tools/ailit/cli.py`](../tools/ailit/cli.py)). Целевое UX: **дефолтный** интерактив — под `ailit agent` (или согласованный alias), без потери `agent run`.

## 4. Этапы и задачи

Нумерация **DP-1 … DP-5** соответствует пяти направлениям из постановки. Каждая задача должна быть взята в работу с явной фиксацией идентификатора (см. workflow).

### Этап DP-1. Глобальный runtime: любая ФС, любой репозиторий

**Цель:** пользователь не обязан «жить» в клоне `ailit-agent`; глобальный конфиг и кэши живут вне репозитория пакета; `project_root` и поиск `.ailit` согласованы с уже закрытым G.3, но **дефолтные пути к секретам** не должны подразумевать `repo_root()` пакета.

#### Задача DP-1.1 — Канон глобального дома `ailit`

**Сделать:** зафиксировать переменную окружения (например `AILIT_HOME` / `AILIT_CONFIG_DIR`) и дефолт `~/.ailit` или XDG `~/.config/ailit` + state/cache по FDO; описать приоритет слоёв в одном модуле (аналог `SETTING_SOURCES` в claude-code).

**Критерии приёмки:** unit-тесты на разрешение пути при заданных env; документированный приоритет global → project → env.

**Референс:** `getClaudeConfigHomeDir` — см. §3.1.

#### Задача DP-1.2 — Install: режим production vs dev

**Сделать:** одна entry-команда (см. этап DP-5) с флагами или подкомандами: **dev** (текущее поведение editable + shim на клон), **install** (копия/wheel в user prefix, shim на стабильный путь).

**Критерии приёмки:** сценарий ручной проверки из README плана: после production-install `ailit` из `$PATH` работает из `/tmp/foo` без `PYTHONPATH`; глобальный конфиг читается.

#### Задача DP-1.3 — Обход проектов из агента

**Сделать:** в интерактивном агенте команды «сменить корень», «открыть репозиторий», отображение текущего корня в шапке; согласование с `ailit config` и TUI.

**Критерии приёмки:** ручной сценарий: два git-клона, переключение без перезапуска процесса (или с явным restart — зафиксировать в постановке задачи).

**Чекпоинт этапа DP-1:** из произвольного каталога виден один и тот же глобальный конфиг; проектные overrides работают.

### Этап DP-2. TUI как основной инструмент `ailit agent` (UX, DeepSeek, меню)

**Цель:** убрать разрыв «TUI не работает с DeepSeek»; сделать TUI местом настройки мира; улучшить chrome (кнопка меню, footer/header); убрать служебные теги из чата; **`ailit agent` без подкоманды** запускает TUI, **`ailit agent run`** — прежний изолированный сценарий.

#### Задача DP-2.1 — Переназначение CLI

**Сделать:** миграция подкоманд и help-текстов; при необходимости оставить `ailit tui` как deprecated alias на один релиз с предупреждением.

**Критерии приёмки:** `ailit agent --help` показывает интерактив и `run`; pytest на парсер CLI (если уже есть паттерн — расширить).

#### Задача DP-2.2 — DeepSeek (и прочие) в TUI наравне с chat

**Сделать:** единый путь загрузки провайдера из merge-конфига в TUI-сессии; устранить расхождение флагов/ограничений, из-за которых TUI сейчас «не тянет» deepseek.

**Критерии приёмки:** ручной сценарий с `DEEPSEEK_API_KEY` (или mock-подстановка): хотя бы один полный turn в TUI; негативный тест при отсутствии ключа — понятное сообщение без traceback.

#### Задача DP-2.3 — Визуальная чистота чата

**Сделать:** заменить `[dim]user>[/dim]` на стилизованные виджеты или `RichLog` с явными `Markup`/`Style` без сырого вывода тегов; единый стиль для user/assistant/tool.

**Критерии приёмки:** скриншотный чеклист в PR или textual snapshot-тест, если введён в репозитории; отсутствие строк `DIM`/`[/dim]` в логе.

**Референс:** изучить, как Claude Code разделяет «диагностический» и «пользовательский» вывод (telemetry vs chat) — `betaSessionTracing.ts` рядом с атрибутами `system_prompt` (для идеи разделения каналов, не для копирования кода).

#### Задача DP-2.4 — Меню и chrome Textual

**Сделать:** переработать кнопку/меню (сейчас воспринимается как «ломающая» UI); согласовать с [`Textual`](https://textual.textualize.io/) паттернами `Header`/`Footer`/`CommandPalette`.

**Критерии приёмки:** визуальное ревью + не менее одного автотеста на наличие ключевых биндингов.

#### Задача DP-2.5 — Настройка глобального конфига из TUI

**Сделать:** slash или модальные экраны: провайдер, модель, лимиты, путь к глобальному дому; вызов существующего `ailit config` API без дублирования бизнес-логики.

**Критерии приёмки:** изменение, сделанное в TUI, видно в `ailit config show` из shell.

**Чекпоинт этапа DP-2:** интерактивный агент по умолчанию в терминале — основной способ работы; DeepSeek рабочий; UI без служебной разметки.

### Этап DP-3. Инкрементальный вывод в `ailit chat` и интерактивном агенте

**Цель:** токены ответа и промежуточные статусы (tool start/end) отображаются по мере поступления; нет ощущения «зависло на 60 секунд → простыня».

#### Задача DP-3.1 — Контракт событий session loop

**Сделать:** узкий внутренний поток событий (аналог подхода `Bus` в opencode): `assistant_delta`, `tool_call_started`, `tool_result`, `usage`, `error` — без привязки UI к деталям провайдера.

**Критерии приёмки:** unit-тесты на reducer/ordering; документ «контракт» в комментарии модуля или `docs/` только если это уже принято в репозитории для подобных вещей.

#### Задача DP-3.2 — Streamlit chat

**Сделать:** подписка на события в [`tools/ailit/chat_app.py`](../tools/ailit/chat_app.py) с обновлением виджетов без полной перерисовки истории.

**Критерии приёмки:** ручной сценарий с медленным провайдером или стабом, демонстрирующий дельты.

#### Задача DP-3.3 — Textual / интерактивный агент

**Сделать:** `call_from_thread` / `post_message` паттерны Textual для безопасного обновления `RichLog` из фона; бэк-прешер на размер буфера.

**Критерии приёмки:** отсутствие гонок (тест или stress-ручник); лимит длины лога настраиваемый.

**Чекпоинт этапа DP-3:** пользователь видит прогресс длинного запроса в обоих UI.

---

### Этап DP-4. Карта скрытых промптов и оптимизация токенов

**Цель:** явная структура системных инструкций (файлы или dataclass-слои), версионирование, отсутствие дублирования между режимами; телеметрия/лог — опционально с хешем промпта (идея из claude-code telemetry), но **не** в пользовательском чате.

#### Задача DP-4.1 — Инвентаризация и «карта»

**Сделать:** таблица: имя фрагмента, условие включения, приоритет, оценка токенов, владелец (core vs project vs plugin); место хранения — каталог `agent_core` или `tools/ailit/prompts/` (решение за задачей, но **одно**).

**Критерии приёмки:** markdown-таблица в этом плане обновляется ссылкой на файл карты в репозитории **или** генерируется скриптом — выбрать один способ и зафиксировать в задаче.

#### Задача DP-4.2 — Слияние слоёв как в референсах

**Сделать:** API уровня `build_effective_system_messages(...)` с приоритетами по образцу `buildEffectiveSystemPrompt` (см. §3.1) и провайдерскими вставками по образцу `system.ts` opencode (см. §3.2).

**Критерии приёмки:** pytest: фикстуры с override, custom, default; длина итогового списка сообщений стабильна при смене только append-слоя.

#### Задача DP-4.3 — Связка с shortlist/compaction

**Сделать:** явные короткие инструкции в компакции не дублируют базовый системный промпт; проверка на «двойной system» в одном turn.

**Критерии приёмки:** регрессионный тест на размер сообщений до/после compaction (грубый порог).

**Чекпоинт этапа DP-4:** разработчик может ответить на вопрос «какие системные куски сейчас в рантайме» без чтения всего кода.

### Этап DP-5. Деплой одной командой и сохранение настроек при обновлении/удалении

**Цель:** из корня репозитория одна команда выполняет production-установку; интерактивный агент может **донастроить** окружение (PATH, ключи, проверка); `pip uninstall` или смена версии **не** удаляет `~/.ailit` без флага; dev-режим сохраняется для контрибьюторов.

#### Задача DP-5.1 — Единая entry-команда

**Сделать:** например `./scripts/ailit-install` или расширение `scripts/install` с подкомандой `prod` / `dev`; идемпотентность.

**Критерии приёмки:** документированная одна строка в README статуса; CI при наличии job'а на install — зелёный.

#### Задача DP-5.2 — Политика данных пользователя

**Сделать:** явный список каталогов: **сохраняются** при upgrade/uninstall (конфиг, кэш сессий по желанию), **удаляются** с пакетом (venv пакета); команда `ailit doctor` или slash в TUI — диагностика путей.

**Критерии приёмки:** таблица в документе рядом с кодом uninstall/post-install hook (если вводится).

#### Задача DP-5.3 — Мастер первого запуска в TUI (опционально)

**Сделать:** при отсутствии глобального конфига — мягкий onboarding (провайдер, ключ не хранить в открытом виде в файле — только env/secret service stub).

**Критерии приёмки:** ручной сценарий «чистый home» до первого успешного запроса.

**Чекпоинт этапа DP-5:** пользователь описывает установку одной командой; обновление не ломает профиль.

## 6. Риски и ограничения

1. **Лицензии:** не копировать код из `claude-code` без совместимости; только идеи и собственная реализация.
2. **Streamlit и long polling:** потоковость в web-чате может отличаться по UX от TUI; допустимы разные транспорты при одном контракте событий.
3. **Секреты:** глобальный конфиг не должен по умолчанию хранить сырой API key в world-readable файле; предпочтение env и будущему secure storage.
4. **Обратная совместимость:** переименование `tui` → `agent` требует ясного deprecation и заметки в release notes.

## 7. Правила закрытия подзадач (git, проверки)

По аналогии с §12 документа [`ailit-global-agent-teams-strategy.md`](ailit-global-agent-teams-strategy.md):

2. Перед коммитом: `pytest`, `flake8` (если затронут Python), ручные сценарии из критериев задачи.
3. При закрытии этапа **DP-x** — обновить таблицу «Статус» в корневом [`README.md`](../README.md) (кратко, без дублирования всего плана).
