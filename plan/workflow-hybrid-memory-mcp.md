# Workflow: hybrid memory и shared MCP-память (ailit-agent)

Документ задаёт **этапы → задачи → критерии приёмки** для внедрения паттерна **hybrid memory** (приватная рабочая память + shared KB / MCP) в экосистему `ailit`. Перед началом итерации агент открывает этот файл и фиксирует, какие задачи закрывает.

Связанный рецепт и ограничения: [`workflow-token-economy-recipe.md`](workflow-token-economy-recipe.md).

## Этап H0. Контракт области памяти и конфигурация

### Задача H0.1 — Модель scopes

**Содержание:** зафиксировать enum уровней `org` | `workspace` | `project` | `agent` | `run`, правила наследования по умолчанию и то, как scope попадает в каждый write/read (заголовок, префикс пути, отдельное поле — на выбор постановки).

**Критерии приёмки:**

- В репозитории есть один канонический текстовый фрагмент (раздел этого файла или ADR), который агент может процитировать без противоречий.
- Указаны 2 примера: «запись разрешена» и «запись запрещена» для каждого уровня.

**Референсы на локальные репозитории:**

- Идея разделения «что помнит агент» vs внешнее API: `/home/artem/reps/letta/README.md` строки 47–62 (`memory_blocks` при создании агента).
- Политика «не только чат»: `/home/artem/reps/hindsight/README.md` строки 20–23.

### Задача H0.2 — Открытые решения постановки

**Содержание:** закрыть вопросы из [`workflow-token-economy-recipe.md`](workflow-token-economy-recipe.md) (орг vs workspace MCP; vault vs SQLite; namespace второй KB).

**Критерии приёмки:**

- В конце этого файла раздел «Открытые вопросы» обновлён: либо ответы, либо явное «отложено до …» с владельцем.

## Этап H1. Транспорт MCP и границы доверия

### Задача H1.1 — Карта инструментов MCP

**Содержание:** перечислить инструменты, которые shared-слой предоставляет агенту (create/search/link), и какие данные **никогда** не проходят через них (секреты, сырой чат — см. рецепт).

**Критерии приёмки:**

- Таблица «инструмент → вход → выход → scope» в этом файле или в `docs/` (если появится отдельная страница — ссылка отсюда).
- Согласованность с [`workflow-token-economy-recipe.md`](workflow-token-economy-recipe.md), раздел «Канонические ограничения».

**Референсы:**

- MCP + `MEMORY_DIR` для vault: `/home/artem/reps/obsidian-memory-mcp/README.md` строки 71–79 (фрагмент конфигурации Claude Desktop).
- MCP-слой для графа: `/home/artem/reps/graphiti/README.md` строки 38–40 (указание на `mcp_server/README.md`).

### Задача H1.2 — Политика прав и изоляция

**Содержание:** описать, кто может писать в `org` vs `project`, нужен ли review-путь для promoted записей.

**Критерии приёмки:**

- Матрица ролей (человек / CI / агент) и действия read/write.
- Нет неоднозначных формулировок вроде «все могут всё».

## Этап H2. Запись в KB: нормализация и происхождение

### Задача H2.1 — Шаблон записи (entity / fact)

**Содержание:** определить минимальные поля: тип сущности, краткое резюме, ссылки на репо/файл/коммит, scope, автор, время.

**Критерии приёмки:**

- Пример записи в markdown (если vault) **или** в JSON-schema (если БД) — один канонический пример в репозитории.
- Пример явно демонстрирует **отсутствие** сырого лога чата.

**Референсы:**

- Markdown + frontmatter + секции: `/home/artem/reps/obsidian-memory-mcp/README.md` строки 29–55.
- Временные факты и эпизоды: `/home/artem/reps/graphiti/README.md` строки 65–70 (определение context graph и validity).

### Задача H2.2 — Происхождение (provenance)

**Содержание:** откуда взялась запись (run id, tool, человек), как помечается устаревание.

**Критерии приёмки:**

- Описан сценарий «факт опровергнут новым коммитом» с шагами обновления KB.

## Этап H3. Retrieval: index-first

### Задача H3.1 — Контракт поиска

**Содержание:** до подмешивания в промпт: запрос → индекс (FTS / граф / эмбеддинги) → top-k → только выбранные фрагменты.

**Критерии приёмки:**

- Псевдокод или диаграмма (mermaid) в этом файле.
- Указано ограничение на max токенов из KB за один turn.

**Референсы:**

- Hybrid retrieval в Graphiti: `/home/artem/reps/graphiti/README.md` строка 55.
- Экономия контекста и index-first через MCP: `/home/artem/reps/context-mode/README.md` строки 36–40 (sandbox, SQLite/FTS5, BM25, выборка релевантного вместо «сырого» дампа в окно).

### Задача H3.2 — Оценка качества retrieval

**Содержание:** минимальный чеклист ручной проверки (3 сценария: hit, partial miss, conflict).

**Критерии приёмки:**

- Три сценария описаны так, что исполнитель без доменного контекста может пройти их и зафиксировать результат.

## Этап H4. Интеграция с `ailit` (код и CLI)

### Задача H4.1 — Конфигурация

**Содержание:** где в `ailit` задаётся список MCP, переменные окружения, профили «dev / prod» для памяти.

**Критерии приёмки:**

- Изменения в коде сопровождаются **pytest** для затронутых модулей и **flake8** для новых/изменённых файлов (см. [`.cursor/rules/project-workflow.mdc`](../.cursor/rules/project-workflow.mdc)).
- Ручной сценарий: запуск CLI с mock-провайдером и проверкой, что вызовы памяти не ломают сессию (конкретная команда — в задаче на момент реализации).

### Задача H4.2 — Наблюдаемость

**Содержание:** логировать факт обращения к памяти (без утечки содержимого в публичные логи по умолчанию).

**Критерии приёмки:**

- Описание уровней логирования и что считается PII.
