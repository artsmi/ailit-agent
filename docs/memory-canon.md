# Канон памяти и экономии контекста (ailit-agent)

Документ фиксирует **целевую модель**: где заканчивается «экономия окна LLM» и начинается «долговременная правда»; как связаны **сессия прогона**, **локальное хранилище на ПК** и **MCP / shared KB** для нескольких машин и групп агентов.

**Статус:** канон постановки для выравнивания ожиданий команды и планов (`plan/workflow-token-economy-recipe.md`, `plan/workflow-hybrid-memory-mcp.md`, `plan/workflow-memory-3.md`). Реализация по этапам плана.

**Связанные документы:** [`INDEX.md`](INDEX.md), [`../context/INDEX.md`](../context/INDEX.md), [`.cursor/rules/project-workflow.mdc`](../.cursor/rules/project-workflow.mdc).

---

## 1. Проблема, которую решает канон

Без явного разделения уровней команда смешивает ожидания:

- **in-memory pager** (`ContextPageStore`, `read_context_page`) — не то же самое, что **долговременная база знаний**;
- **сжатие в одном `run()`** — не гарантирует **доступ к полному тексту** в **следующем** `run()` или после **рестарта процесса**;
- **MCP / KB** — не заменяет **локальные** файлы проекта и **не** отменяет **pager** внутри прогона.

Канон задаёт: **что считается источником правды** на каждом уровне и **как уровни стыкуются**.

---

## 2. Трёхуровневая модель (сводка)

| Уровень | Область | Что «экономит» | Роль в «финальном каноне» |
|--------|---------|----------------|----------------------------|
| **1** | Сессия `agent run`, `ailit chat`, один вызов `SessionRunner.run()` | Сообщения **в окне** на время прогона: pager, budget, prune, compaction, лимиты, stream | **Экономия окна**; **краткоживущий** `page_id` в RAM (`ContextPageStore`). **Не** единственная долгая память. |
| **2** | Локальное хранилище на ПК, где запущен ailit (репозиторий, `.ailit`, логи, индексы, мультиагенты **на одной машине**) | Не держать в промпте то, что уже **в файлах/индексе**; **догрузка** по id / range / пути | **Локальный канон** проекта и артефактов; **мост** после рестарта процесса **без** обязательного облака. |
| **3** | MCP-сервер(а), общий для нескольких ПК пользователя; **группы** мультиагентов | Вынести **смысл** и **длинные** тела **из** чата; **index-first** retrieval; scopes, provenance | **Межмашинный / межсессионный** смысловой слой; **нормализованные** записи, **не** сырой дамп чата по умолчанию. |

**Формула:** **уровень 1** = тактическая экономия **окна**; **уровни 2 + 3** = стратегическая **память** и **переиспользование «всерьёз»** (проект, команда, сеть).

---

## 3. Уровень 1 — сессия агента, чат, `agent run`

### 3.1. Назначение

Уложиться в **токеновый бюджет** и **длину контекста** **в рамках одного** прогона `run()` (внутри него может быть много внутренних turn’ов до FINISHED / cap / budget).

### 3.2. Механизмы (как в `agent_core` / token-economy recipe)

- **Context pager** — большой вывод инструмента → превью в `messages` + `page_id` + полный текст в **`ContextPageStore` (in-memory)**. Инструмент **`read_context_page`** читает **из этого store**.  
  **Ограничение (текущая реализация):** в начале каждого `run()` store **очищается** (если включены pager/budget). Новый экземпляр `SessionRunner` + новый `run()` (как в типичном сценарии **каждого** user-сообщения в `ailit chat`) **не** сохраняет полные страницы **между** user-turn’ами, пока **явно** не введён bridge на уровень 2/3.
- **Tool output budget** — лимит суммарного размера батча; при необходимости **дополнительно** уводит часть в «страницы».
- **Tool output prune** — укорочение **уже накопленной** истории сообщений по политике; **в** итоговый список `messages` попадает уже **урезанный** текст → эффект **сохраняется** в истории, которую читает следующий `run()`.
- **Compaction / лимиты turn’ов** — сжатие хвоста, лимит шагов, отдельные настройки сессии.
- **BudgetGovernance (usage)** — лимиты на сессию; **косвенно** влияет на размер контекста.

### 3.3. «Финальный канон» на уровне 1

- **Канон экономии *контекстного окна* в run:** pager + budget + prune + compaction + сессионные лимиты + (при необходимости) stream-редуктор.
- **Канон *долговременной* полноты текста** уровень 1 **не** определяет: полный текст «страницы» **живёт** в RAM **только** до логики окончания/начала `run()` и **очистки** store.

### 3.4. Связь с уровнями 2 и 3

- **1 не подменяет 2/3.** Нужен полный текст **в следующем** `run()` или **после** рестарта → **догрузка** с **2** (файл, кэш, индекс, артефакт) или **3** (MCP: search / fetch), либо **повтор** исходного tool-вызова.
- **1 может вызывать 3:** если в реестр подмешаны MCP-инструменты, **экономия окна** = короткие аргументы и ответы, **без** вставки сырых мегабайт в `messages`.
- **1 может писать в 2** через обычные project tools (`write_file`, артефакты, mailbox): тогда **канон** для последующих чтений — **артефакт на диске** (уровень 2), а не «вечный» `page_id` в RAM.

## 4. Уровень 2 — локальное хранилище на ПК (мультиагенты, один workspace)

### 4.1. Назначение

- **Пережить** остановку процесса ailit.
- **Согласовать** несколько агентов / сессий, работающих с **одной** копией репозитория и **локальными** путями.
- Дать **index-first** доступ: **сначала** указатель / поиск / top-k, **потом** догрузка.

### 4.2. Типовое содержимое (в духе `ai-multi-agents` и планов ailit)

- **Канон в репозитории:** структурированные каталоги вроде `context/*` (архитектура, протоколы, тесты, человеко-читаемая память) — **файлы как source of truth** (см. донорский паттерн «markdown canonical + derived index»).
- **Производные индексы:** SQLite / FTS / эмбеддинги — **ускорители** и **пересчитываемы**; **не** единственная «правда» в отрыве от файлов.
- **Артефакты и логи:** JSONL-диагностика (например `ailit-*.log`), служебные каталоги (`.ailit` и т.д.), **без** подмены **семантического** shared KB **необработанным** чат-логом.
- **Mailbox / team-артефакты** — **координация** мультиагентов **на одной** машине/клоне.

### 4.3. Экономия контекста

- Паттерн **index → (top-k / id) → read fragment**, аналог **ContextPackBuilder** (ruflo) и **ctx_search + выборка** (context-mode).
- **Тактический bridge «тот же `page_id`»:** хранение полного тела на диске (или в локальном blob-store) как артефакта, связанного с `page_id` / run id, и подстановка в `ContextPageStore` при старте `run()`, если такая постановка принята. Это уровень 2, не расширение чисто RAM без политики TTL/очистки.

### 4.4. «Финальный канон» на уровне 2

- **Локальный** канон **проектных** знаний и **артефактов**, доступный **всем** сессиям ailit на **этом** ПК, с **отчётом** в git/ревью по правилам проекта.
- **Сеть не обязательна**; при появлении **уровня 3** **2** остаётся **кэшем/зеркалом/рабочей копией** по согласованной политике.

---

## 5. Уровень 3 — MCP, несколько ПК, группы мультиагентов

### 5.1. Назначение

- **Общий** смысловой слой: **org / workspace / project** scopes, **provenance**, **нормализованные** сущности и факты.
- **Index-first** retrieval: **в** модель — только **отобранные** фрагменты.
- **Политика доверия:** кто **куда** пишет, изоляция, **без** публикации PII/секретов по умолчанию.

### 5.2. Типовые «функции уровня» (абстрактно)

- Create / search / link записей (пример: vault markdown + frontmatter, граф сущностей, **hybrid** retrieval).
- **Не** везти **весь** сырой output tools в **KB** — только **нормализованные** и **осмысленные** с точки зрения постановки (см. ограничения `workflow-hybrid-memory-mcp.md`).

### 5.3. Экономия контекста

- **Длинный** материал **хранится** на стороне MCP/хранилища; **в** окно — **короткие** выбранные куски.
- **Временнáя** модель (как в graph-ориентированных донорах) снижает повтор **устаревшей** простыни **в** промпте.

### 5.4. «Финальный канон» на уровне 3

- **Канон** для **межсессионного** и **межмашинного** **смысла** и **групповой** работы, при **согласованных** **scopes** и **governance** (см. этапы H* в hybrid workflow, M3 в `workflow-memory-3.md`).

### 5.5. Когда уровень 1 использует **логику** уровня 3 на одной машине

- MCP **процесс** может крутиться **локально**; **с точки зрения** экономии контекста **механика** та же: **сначала** **инструмент** поиска/выборки, **короткий** ответ **в** `messages`.  
- **Сеть** **не** обязательна, чтобы **иметь** **паттерн** «данные **вне** окна, **догрузка** **по** **запросу**».

### 5.6. Соотношение 2 и 3

- **2** — локальные файлы и клон репо; **3** — общий слой для нескольких ПК / команды. Дубли избегают чёткими ролями: что только локально, что только в MCP, что синхронизируется явно (документируется в постановке hybrid и deploy).

## 6. Зависимости уровней (сводка)

```
Уровень 3 (MCP / shared KB)  ──опционально──►  Уровень 1 (tools в run)
        ▲                                              │
        │  политика write / read                       │ короткие вызовы
        │                                              ▼
Уровень 2 (файлы, индексы, артефакты на ПК) ◄────  материализация / bridge
        ▲
        │  персист, git, mailbox
  Долгоживущая «правда» для run N+1
```

- Без уровней **2** и **3**: **1** сжимает окно, но не обещает долгую полноту по «тому же» `page_id` в RAM.
- С **2** и по необходимости **3**: разделять рабочий эфемерный слой (в т.ч. pager в run) и то, что сохранено как знание/артефакт с политикой.

## 7. Методики (глоссарий для сопоставления с донорами)

| Методика | Суть | Типичный уровень в каноне ailit |
|----------|------|---------------------------------|
| **A. Эфемерный runtime (pager / превью в run)** | Полный кусок не в окне целиком; догрузка из RAM store в пределах жизни store | **1** |
| **B. Index-first, fetch-by-id** | Сначала индекс или указатели, затем фрагмент | **1** + **2** (инструменты читают с диска) |
| **C. Durable KB + tools (MCP и аналоги)** | Смысл вне чата, доступ через поиск/fetch, scopes | **3** |
| **D. Канон = файлы + derived index** | Правда в репо; БД — кэш/ускоритель | **2** (как в ai-multi-agents и сходных паттернах) |
| **E. Слои памяти (working / …)** | Working — краткоживущее; долгое — отдельно | **1** vs **2+3**; детализация в M3 |

**Тактический bridge** (тот же `page_id` → артефакт на уровне 2) — отдельное соглашение; он не заменяет канон C/D, если нужны review, scopes и групповой доступ.

## 8. Соотнесение с репозиториями-донорами (логическое, не 1:1)

Таблица не копирует код: фиксируется, какой *тип паттерна* у донора и к какому уровню канона ailit он ближе. Подробнее — в [`INDEX.md`](INDEX.md), раздел «Локальные репозитории-референсы» и [`.cursor/rules/project-workflow.mdc`](../.cursor/rules/project-workflow.mdc).

| Репозиторий (см. также workflow-док) | Идеи донора | Уровни канона ailit |
|----------------------------------------|-------------|---------------------|
| **context-mode** | Сырьё вне окна; sandbox; continuity через SQLite+FTS и релевантный fetch, а не вся сырая история в чат | **1** (экономия) + паттерн **2** (index→fetch на машине); по смыслу близко к **3**, если тот же контракт вынесен в MCP |
| **ruflo** (`plan/03_token_optimization` и т.д.) | TokenBudgetManager, ContextPackBuilder (index → fetch → full), ToolExposurePolicy, SemanticResponseCache | **1** (governance окна) + методика **B** на **1–2** |
| **ai-multi-agents** | Четыре слоя памяти; `context/*` как канон; sqlite — derived; rules — procedural; workspace на одной машине | **2** (файлы + индексы) и **E**; **1** = working-часть в рантайме ailit |
| **graphiti** | Temporal context graph, episodes как provenance, hybrid retrieval | **3** (глубина + время), **C**; **2** = копия/клиент у потребителя |
| **obsidian-memory-mcp** | Сущности как markdown, `MEMORY_DIR`, MCP, `[[links]]` | **3** (vault + инструменты), частично **D** по духу файлов |
| **hindsight, letta** | Долговременная / блочная память через API, не только чат | вдохновляют **2–3**, **не** заменяют pager **1** |
| **claude-code, opencode** | Compaction, loop, events, streaming, session model | в основном **1** + наблюдаемость; связь с **2** через логи/артефакты |
| **ai-multi-agents** (как оболочка разработки) | Проекты, `context/`, мультиагентные сценарии | **2** (локальный workspace) + стык **1** с ailit |

**Вывод для команды:** доноры *согласуемы* с каноном, если **не** ожидать, что **pager (A)** *одновременно* и долгоживущая вся-полнота-текста. У доноров «долгая память» **— это B/C/D** через **2** и **3** (как в планах M3 и hybrid).

## 10. Практичные правила для команды (чек-лист)

1. Сэкономить токены *сейчас* в `run` — **уровень 1**.
2. Снова получить полноту *после* стопа/рестарта — **2** (файл, индекс) и/или **3** (MCP).
3. Один `page_id` из *предыдущего* user-сообщения в чате — *сейчас* требует bridge, повтор tool или MCP, не «сам по себе» pager.
4. Не смешивать «канон KB (3)» и «сырой лог сессии» без нормализации.
5. Мультиагенты на одной машине опираются на **2**; разные ПК/группы — на **3**, если нужен общий смысловой слой, не только git.
