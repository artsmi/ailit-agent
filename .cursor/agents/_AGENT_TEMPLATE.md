# Шаблон единого файла агента

Этот файл не является исполняемым агентом. Он задаёт целевую структуру для
поэтапного переноса правил из `.cursor/rules/system/**` в самодостаточные
файлы `.cursor/agents/NN_role.md`.

Цель миграции: каждый агент содержит всю информацию о своей роли, процессе,
артефактах, JSON-форматах, примерах и anti-patterns. Внешними ссылками в начале
остаются только проектные правила из `.cursor/rules/project/**` и выбранные
канонические файлы `context/*`, если они нужны роли как данные проекта.

---

## 1. Frontmatter

Каждый исполняемый агент начинается с YAML frontmatter:

```yaml
---
name: role_name
description: Короткое описание роли, результата и главного артефакта.
---
```

Требования:

- `name` совпадает с именем роли без номера, например `developer`.
- `description` короткий: роль, действие, основной результат.
- Не добавляй ссылки на `.cursor/rules/system/**` в frontmatter.

---

## 2. Заголовок И Назначение

Формат:

```markdown
# Название роли (NN)

Ты — <роль>. Твоя задача: <1-3 предложения>.
```

В этом разделе агент должен сразу понимать:

- за какой этап pipeline он отвечает;
- какой главный результат он производит;
- какую работу он точно не выполняет;
- кто потребляет его результат.

Пример:

```markdown
# Разработчик (08)

Ты реализуешь одну задачу из `context/artifacts/tasks/task_X_Y.md`, запускаешь
проверки по этой задаче и возвращаешь JSON-ответ 08 плюс test report.
Ты не перепланируешь scope, не проводишь code review и не обновляешь
долговременный `context/*` вместо writer pipeline.
```

---

## 3. Project Rules В Начале Файла

В начале каждого агента оставляются только ссылки на проектные правила:

```markdown
## Project Rules

Прочитай только те проектные правила, которые применимы к текущему входу:

- [`../rules/project/project-config.mdc`](../rules/project/project-config.mdc)
- [`../rules/project/project-agent-models.mdc`](../rules/project/project-agent-models.mdc) — только для оркестратора или если выбирается модель агента.
- [`../rules/project/project-orchestrator-overrides.mdc`](../rules/project/project-orchestrator-overrides.mdc) — только для `01_orchestrator`.
- [`../rules/project/project-workflow.mdc`](../rules/project/project-workflow.mdc) — если задача касается commit/workflow проекта.
- [`../rules/project/project-code-python.mdc`](../rules/project/project-code-python.mdc) — если diff затрагивает Python.
- [`../rules/project/project-code-c.mdc`](../rules/project/project-code-c.mdc) — если diff затрагивает C.
- [`../rules/project/project-code-cpp.mdc`](../rules/project/project-code-cpp.mdc) — если diff затрагивает C++.
- [`../rules/project/project-code-review.mdc`](../rules/project/project-code-review.mdc) — для `09_code_reviewer`.
- [`../rules/project/project-architecture-review.mdc`](../rules/project/project-architecture-review.mdc) — для `05_architecture_reviewer`.
```

Правило миграции:

- Не ссылаться из агента на `.cursor/rules/system/**`.
- Если агенту нужны бывшие system rules, перенеси их смысл прямо в тело агента.
- Если правило относится к конкретному проекту, оставь его в `.cursor/rules/project/**` и ссылайся на него здесь.

## 4. Роль И Границы

Раздел фиксирует ownership роли:

```markdown
## Роль И Границы

Ты делаешь:

- ...

Ты не делаешь:

- ...

Границы ответственности:

- Вход от предыдущей роли: ...
- Выход для следующей роли: ...
- При конфликте входных данных: ...
```

Обязательно указать:

- что агент имеет право менять или создавать;
- какие артефакты он не трогает;
- когда он обязан остановиться и вернуть blocker;
- какие решения он не принимает самостоятельно.

---

## 5. Входные Данные

Формат:

```markdown
## Входные Данные

Ожидаемый вход от оркестратора:

- `artifacts_dir`: ...
- основной файл этапа: ...
- предыдущие артефакты: ...
- код / diff / test logs: ...
- ограничения пользователя: ...

Если вход неполный:

1. Не продолжай по догадке.
2. Верни `has_open_questions` / `blocked` по формату роли.
3. Перечисли, каких данных не хватает и какой шаг они блокируют.
```

Для каждого агента нужно перечислить конкретные сценарии входа. Например для
`08_developer`: новая задача, `fix_by_review`, `fix_by_tests`.

---

## 6. Политика Чтения Контекста

Раздел заменяет отдельные read-policy system rules:

```markdown
## Политика Чтения Контекста

Порядок чтения:

1. Прочитай применимые project rules из раздела `Project Rules`.
2. Прочитай входной артефакт текущей роли.
3. Прочитай только нужные индексы `context/*`.
4. Дочитай полные файлы только если они нужны для решения текущей задачи.

Запрещено:

- читать весь `.cursor/rules/system/**`;
- читать весь `context/` "на всякий случай";
- подмешивать старые review-итерации, если нужен только последний артефакт;
- заменять канонический `context/*` результатами semantic search или локального индекса.
```

Если роль всегда работает с определёнными индексами, укажи их явно:

- `context/arch/INDEX.md`;
- `context/proto/INDEX.md`;
- `context/start/INDEX.md`;
- `context/tests/INDEX.md`;
- `context/memories/index.md`.

## 7. Процесс Работы

Главный раздел агента. Он должен содержать полный алгоритм без ссылок на
system rules.

Формат:

```markdown
## Процесс Работы

### Сценарий A: ...

1. ...
2. ...
3. ...

### Сценарий B: ...

1. ...
2. ...
3. ...
```

Для review-ролей обязательно описать:

- что проверять;
- как классифицировать замечания;
- когда `approved`, `rework_required`, `blocked`, `rejected`;
- почему upstream `passed` не равен approval.

Для developer/test ролей обязательно описать:

- какие проверки запускать;
- что считать `failed`;
- что считать `blocked_by_environment`;
- почему blocked не равен passed.

Для orchestrator обязательно описать:

- state machine pipeline;
- порядок ролей;
- review loops;
- `task_waves`;
- parallel barriers;
- completion gate;
- правила эскалации пользователю.

---

## 8. Артефакты И Пути

Раздел заменяет `artifact-*.mdc` для конкретной роли.

```markdown
## Артефакты И Пути

Ты создаёшь или обновляешь:

- `{artifacts_dir}/...`

Ты читаешь:

- `{artifacts_dir}/...`

Ты не создаёшь:

- ...
```

Для каждого файла указать:

- кто producer;
- кто consumer;
- обязательный или optional;
- когда файл считается валидным;
- какие старые файлы нельзя использовать как источник истины.

## 9. Машиночитаемый Ответ

Каждый агент, который возвращает JSON, содержит схему прямо в своём файле.

Формат:

```markdown
## Машиночитаемый Ответ

Ответ всегда начинается с JSON:

```json
{
  "field": "value"
}
```

Поля:

- `field`: допустимые значения, семантика, правила согласованности.

Правила:

- Если `approval_blockers` не пустой, итог не может быть `approved`.
- Если required evidence `missing` / `blocked` / `failed`, итог не может быть `approved`.
```

Важно:

- Схема должна быть самодостаточной.
- Примеры не должны противоречить описанию полей.
- Если роль не возвращает JSON, явно напиши это.

---

## 10. Markdown-Отчёт

Если после JSON нужен текстовый отчёт, структура задаётся прямо в агенте:

```markdown
## Markdown-Отчёт

После JSON верни краткий markdown:

1. Итоговое решение.
2. Что проверено / сделано.
3. Тесты и evidence.
4. Blockers / open questions.
5. Следующий шаг для оркестратора.
```

Для review-ролей findings идут первыми:

- `BLOCKING`;
- `MAJOR`;
- `MINOR`;
- open questions;
- summary.

## 11. Статусы И Gate-Семантика

В каждом агенте нужно явно описать статусы именно этой роли.

Пример:

```markdown
## Статусы И Gate-Семантика

- `passed`: все обязательные проверки выполнены и прошли.
- `failed`: команда дошла до кода приложения и выявила дефект.
- `blocked_by_environment`: проверка не смогла стартовать или дойти до кода из-за внешнего условия.
- `missing`: обязательное evidence не было выполнено и не имеет valid external blocker.

Правила согласованности:

- `blocked_by_environment` не равен `passed`.
- Финальный completion не может скрывать незакрытый required evidence.
- Локальный `passed` test runner не заменяет code review approval.
```

Для ролей `09` и `01` этот раздел критичен: он должен исключать ситуацию,
когда pipeline закрывается без обязательного review или live evidence.

---

## 12. Blockers И Open Questions

Раздел описывает, когда агент обязан остановиться:

```markdown
## Blockers И Open Questions

Остановись и верни blocker, если:

- входные артефакты противоречат друг другу;
- required evidence невозможно получить;
- задача требует выбора архитектурного контракта, которого нет в ТЗ/архитектуре/плане;
- runtime tool или Subagent type недоступен;
- реализация требует выйти за scope.

Формат вопроса:

1. Контекст.
2. Проблема.
3. Варианты решения.
4. Что блокируется.
5. Какой ответ нужен от пользователя.
```

---

## 13. Тесты И Evidence

Для ролей, связанных с кодом или проверками:

```markdown
## Тесты И Evidence

Обязательные проверки:

- новые тесты, добавленные или изменённые в задаче;
- exact tests из task-файла;
- минимальный регресс по затронутому path;
- e2e/smoke для production-relevant веток.

Evidence rules:

- Не выдавай fake model, mock provider, stub runtime или harness за production-like evidence.
- Если live evidence требуется, выполни его или честно верни `blocked_by_environment`.
- Если команда дошла до кода и упала, это `failed`, а не `blocked_by_environment`.
```

Если агент не запускает тесты, явно напиши:

```markdown
Эта роль не запускает тесты. Она только проверяет / планирует / описывает,
какие проверки должны выполнить последующие роли.
```

## 14. Примеры

Каждый агент должен содержать хорошие и плохие примеры для своей роли.

Формат:

```markdown
## Примеры

### Хороший пример

...

Почему хорошо:

- ...

### Плохой пример

...

Почему плохо:

- ...
```

Для JSON-ролей нужны минимум:

- пример успешного ответа;
- пример blocked/rework/failed ответа;
- пример конфликта входных данных.

Для developer-роли нужны минимум:

- пример отчёта с passed checks;
- пример `blocked_by_environment`;
- пример `fix_by_review` без расширения scope.

---

## 15. Anti-Patterns

Формат:

```markdown
## Anti-Patterns

Запрещено:

- ...
```

Anti-patterns должны быть конкретными для роли.

Общие anti-patterns:

- ссылаться на `.cursor/rules/system/**` вместо встроенного текста;
- закрывать обязательный gate статусом `passed_with_gap` без явной семантики;
- объявлять completion без обязательного артефакта роли;
- подменять review тестовым отчётом;
- подменять live evidence unit-тестом, если task contract требует product path;
- расширять scope вместо возврата blocker.

## 16. Checklist

Финальный самоконтроль агента:

```markdown
## Checklist

- [ ] Прочитаны применимые project rules.
- [ ] Прочитан входной артефакт текущего этапа.
- [ ] Не использованы `.cursor/rules/system/**`.
- [ ] Выполнены только действия своей роли.
- [ ] Обязательные артефакты созданы или явно заблокированы.
- [ ] JSON соответствует схеме этого файла.
- [ ] Blockers / missing evidence не замаскированы под success.
- [ ] Следующий шаг для оркестратора понятен.
```

## 17. Миграционный Чеклист Для Существующего Агента

Для переноса конкретного `NN_role.md`:

1. Скопировать этот шаблон в рабочий draft роли.
2. Оставить вверху только применимые `.cursor/rules/project/**`.
3. Найти все `.cursor/rules/system/**`, на которые сейчас ссылается агент.
4. Перенести содержимое этих system rules в соответствующие разделы агента:
   - process rules → `Процесс Работы`;
   - artifact schemas → `Артефакты И Пути` и `Машиночитаемый Ответ`;
   - test rules → `Тесты И Evidence`;
   - completion/status rules → `Статусы И Gate-Семантика`;
   - examples → `Примеры`;
   - anti-patterns/checklists → одноимённые разделы.
5. Удалить ссылки на `.cursor/rules/system/**`.
6. Проверить, что агент можно понять без открытия других system-файлов.
7. Проверить, что project-specific правила не задублированы в агенте, а остались ссылками.
8. Сверить, что примеры не противоречат JSON-схеме и gate-семантике.
9. Для `01_orchestrator` отдельно проверить completion gate:
   - есть все обязательные артефакты;
   - `09_code_reviewer` не пропущен;
   - required evidence `blocked/missing/failed` не может стать `approved`;
   - финальный `11` не скрывает незакрытые gaps.

## 18. Рекомендуемый Порядок Миграции Ролей

1. `00_agent_development` — общий канон и терминология.
2. `09_code_reviewer` — gate-семантика approval и missing evidence.
3. `11_test_runner` — статусы `passed` / `failed` / `blocked_by_environment`.
4. `01_orchestrator` — completion gate, обязательность `09`, blockers.
5. `08_developer` — task execution, reports, fix_by_review/fix_by_tests.
6. `02`-`07` — ТЗ, review ТЗ, архитектура, review архитектуры, план, review плана.
7. `12_change_inventory` и `13_tech_writer` — writer pipeline.
8. `10_researcher` — отдельный research-flow.

Такой порядок сначала устраняет проблему ложного completion, а потом переводит
остальные роли на самодостаточные prompt-файлы.
